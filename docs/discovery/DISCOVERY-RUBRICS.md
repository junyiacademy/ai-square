# Discovery 設計規範與評分標準

> 所有 Discovery 職業 YAML 的品質標準、任務設計原則、AI 批改準則
> 建立新職業或修改現有內容時必須遵循

---

## 一、Discovery 的本質

Discovery 不是考試，不是教學影片，是**沉浸式職涯體驗**

核心理念：
- 讓 15-18 歲的學生**扮演**某個職業的新人
- 透過**解決真實工作場景的問題**來認識這個職業
- 每個任務都是一個「工作中會遇到的情境」，不是課本習題
- AI 扮演的是「同事/導師」，不是「老師/考官」

### Discovery 的 5 個設計原則

| 原則 | 說明 | 反例（不要做） |
|------|------|--------------|
| **沉浸感** | 學生忘記自己在「上課」，覺得自己在「工作」 | 「請回答以下問題」 |
| **具體性** | 每個任務都有明確的產出物 | 「了解 XX 的概念」 |
| **漸進性** | 從觀察 → 分析 → 動手 → 創造 | 第一個任務就要設計完整系統 |
| **在地性** | 連結台灣產業、企業、現況 | 只講美國矽谷 |
| **回饋性** | 做完馬上知道做得如何、為什麼 | 只給分數沒有解釋 |

## 二、職業 YAML 品質標準（Career Rubrics）

每個職業 YAML 必須達到以下標準，滿分 50 分

### 2.1 世界觀（world_setting）— 10 分

| 分數 | 標準 |
|------|------|
| 9-10 | 世界觀有獨特名稱、明確的氛圍、讓人想探索的設定。能在一句話內讓學生「看到」這個地方 |
| 7-8 | 有名稱和描述，但偏通用，缺少讓人記住的細節 |
| 5-6 | 有描述但感覺像教科書，不像一個「地方」 |
| 0-4 | 沒有世界觀或只是一行文字 |

**檢查項目**：
- [ ] 世界觀名稱是否有記憶點？（「機甲工坊」>「機器人實驗室」）
- [ ] 描述是否讓人能「畫面化」？
- [ ] 是否跟台灣產業有連結？

### 2.2 起始場景（starting_scenario）— 15 分

| 分數 | 標準 |
|------|------|
| 13-15 | 5 要素齊全（角色/背景/NPC/任務目標/成功標準），情境緊迫有張力，學生讀完就知道要做什麼 |
| 10-12 | 5 要素大致齊全，但任務目標不夠具體，或成功標準太模糊 |
| 7-9 | 缺少 1-2 個要素，或情境太平淡沒有張力 |
| 0-6 | 只有一段描述，缺少明確的任務指引 |

**5 要素檢查**：
- [ ] **角色**：你是誰？職稱 + 你在組織裡的位置（「見習技師」不是「工程師」）
- [ ] **背景**：為什麼是今天？（「第一天上班」「剛通過考試」「被緊急召集」）
- [ ] **NPC**：導師/主管的名字 + 跟你的關係 + 他的態度（嚴肅？溫暖？焦急？）
- [ ] **【你的任務】**：用 4 個字以內的動詞開頭（「診斷」「分析」「設計」「找出」），具體說明要產出什麼
- [ ] **【成功標準】**：學生讀完知道「做到什麼程度算好」

### 2.3 技能樹（skill_tree）— 10 分

| 分數 | 標準 |
|------|------|
| 9-10 | 4 個核心技能涵蓋職業的關鍵面向，進階技能有合理的前置條件，技能名稱用學生聽得懂的詞 |
| 7-8 | 技能結構合理但名稱太專業，或前置條件不合理 |
| 5-6 | 技能太少或太多，結構不清楚 |
| 0-4 | 沒有技能樹或技能跟職業不相關 |

**檢查項目**：
- [ ] 核心技能是否涵蓋：技術面 + 思維面 + 溝通面 + 產業面？
- [ ] 技能名稱是否讓 15 歲學生也能理解？
- [ ] 進階技能的 requires 是否合理？（A 學會才能學 B）

### 2.4 里程碑（milestone_quests）— 10 分

| 分數 | 標準 |
|------|------|
| 9-10 | 3 個里程碑從簡單到困難，每個都是一個完整的「專案」而非單一題目，獎勵有意義 |
| 7-8 | 有里程碑但難度差異不大，或獎勵沒有吸引力 |
| 5-6 | 里程碑描述模糊，不清楚要做什麼 |
| 0-4 | 沒有里程碑或只有一個 |

### 2.5 在地連結（台灣產業）— 5 分

| 分數 | 標準 |
|------|------|
| 5 | 情境中自然提到台灣企業/產業/法規/現況（不是硬塞） |
| 3-4 | 有提到但只是一句話帶過 |
| 1-2 | 完全沒有在地連結，情境可以發生在任何國家 |

## 三、任務內容品質標準（Task Rubrics）

每個任務（包含 AI 生成的 adaptive tasks）必須符合

### 3.1 任務描述 — 必須包含

```
1. 情境（什麼事情發生了）
2. 你的角色（你在這件事裡扮演什麼）
3. 具體要求（你要產出什麼：報告/方案/分析/設計）
4. 限制條件（時間/資源/範圍的約束）
5. 評判標準（怎樣算做得好）
```

### 3.2 任務類型與認知層次

每個任務應對應 Bloom's Taxonomy 的某個層次，且從低到高排列：

| 順序 | 認知層次 | 任務類型 | 動詞 | 範例 |
|------|---------|---------|------|------|
| 1 | 理解 | chat | 解釋、描述、比較 | 「用你自己的話解釋退相干」 |
| 2 | 應用 | creation | 計算、使用、示範 | 「設計一個感測器配置方案」 |
| 3 | 分析 | analysis | 比較、歸類、診斷 | 「分析不良品率的三個可能原因」 |
| 4 | 評估 | evaluation | 評估、判斷、辯護 | 「評估兩個方案的優劣並選擇」 |
| 5 | 創造 | creation | 設計、規劃、提案 | 「設計完整的自動化生產線」 |

**規則**：
- 歡迎任務（第一個）必須是「理解」或「應用」層次
- 不能跳過層次（不能第一個就要「創造」）
- 每個 program 的 4 個任務應涵蓋至少 3 個不同層次

### 3.3 好任務 vs 壞任務

| 好任務 | 壞任務 |
|--------|--------|
| 「診斷這台機器人的感測器故障，提出修復方案」 | 「了解感測器的基本概念」 |
| 「為這個 App 的崩潰問題寫一份 bug report」 | 「學習 debugging 技巧」 |
| 「評估太陽能和風力在台南的效率差異」 | 「了解再生能源」 |
| 「設計一個讓 3% 不良品率降到 0.5% 的改善方案」 | 「學習品質管理」 |

關鍵差異：好任務有**具體產出物**，壞任務只有**抽象學習目標**

## 四、AI 批改準則（Evaluation Rubrics）

AI 評估學生回答時，必須依據以下框架

### 4.1 評分維度（4 維度，各 25 分，滿分 100）

#### 維度 1：理解力（25 分）
學生是否理解問題的本質？

| 分數 | 標準 |
|------|------|
| 21-25 | 準確辨識問題核心，展現對領域知識的理解，能連結相關概念 |
| 16-20 | 理解問題但有些概念不夠精確 |
| 11-15 | 部分理解，有明顯的知識缺口 |
| 6-10 | 理解表面但抓不到核心 |
| 0-5 | 完全偏離題目或沒有展現理解 |

#### 維度 2：分析力（25 分）
學生是否有條理地分析問題？

| 分數 | 標準 |
|------|------|
| 21-25 | 有系統地拆解問題，考慮多個面向，提出有邏輯的推論 |
| 16-20 | 有分析但面向不夠全面 |
| 11-15 | 分析缺乏結構，跳躍式思考 |
| 6-10 | 只描述現象沒有分析 |
| 0-5 | 沒有分析 |

#### 維度 3：實踐力（25 分）
學生的方案是否可行？

| 分數 | 標準 |
|------|------|
| 21-25 | 方案具體可行，考慮了限制條件和可能的風險 |
| 16-20 | 方案大方向正確但缺少執行細節 |
| 11-15 | 方案太籠統或不太可行 |
| 6-10 | 只有想法沒有方案 |
| 0-5 | 沒有提出方案 |

#### 維度 4：表達力（25 分）
學生是否清楚地表達了自己的思路？

| 分數 | 標準 |
|------|------|
| 21-25 | 條理清晰、邏輯連貫，用詞精準 |
| 16-20 | 大致清楚但有些地方需要更好的組織 |
| 11-15 | 能看出想法但表達混亂 |
| 6-10 | 很難理解學生想表達什麼 |
| 0-5 | 無法理解或答非所問 |

### 4.2 回饋撰寫原則

AI 的回饋必須遵循：

1. **先肯定再建議**：找到學生做對的地方，真誠地肯定（不是敷衍的「做得不錯」）
2. **具體指出問題**：不說「需要更深入」，說「你提到了感測器但沒有說明用哪種類型、為什麼選它」
3. **給出下一步**：不只是指出問題，要告訴學生「如果要改善，你可以...」
4. **角色代入**：用導師/同事的口吻，不是考官的口吻。「陳師傅看了你的方案會說...」
5. **連結學習目標**：「這個分析展現了你對 XX 的理解，接下來你需要加強 YY」

### 4.3 評分等級對應

| 總分 | 等級 | 回饋語氣 | XP 比例 |
|------|------|---------|---------|
| 85-100 | 優秀 | 導師很欣賞你的表現 | 100% |
| 70-84 | 良好 | 方向正確，有幾個地方可以更好 | 80% |
| 55-69 | 需改進 | 看到你的努力，但需要在 XX 方面加強 | 60% |
| 40-54 | 待加強 | 基礎概念需要釐清，建議先 XX | 40% |
| 0-39 | 重新嘗試 | 可能偏離了任務方向，讓我們重新看看任務要求 | 20% |

### 4.4 禁止的 AI 回饋行為

- **永遠不要**：「你的回答不夠好」「你沒有理解」「這是錯的」
- **永遠不要**：只給分數沒有解釋
- **永遠不要**：批評學生「沒有回應任務指示」（如果任務指示不夠清楚，是設計的問題不是學生的問題）
- **永遠不要**：用跟學生回答語言不同的語言回饋
- **永遠不要**：給出超過 3 個改進建議（太多會打擊信心）

## 五、YAML 建立檢查清單

新建或修改任何 Discovery 職業 YAML 前，必須通過以下檢查：

### 必檢項目

```
□ metadata.title 是否有記憶點？（不是通用的「XX 工程師」）
□ metadata.short_description 是否一句話就能引起好奇？
□ world_setting.name 是否像一個「地方」而不是「概念」？
□ starting_scenario 是否包含 5 要素（角色/背景/NPC/任務/標準）？
□ starting_scenario 的任務是否有具體產出物？
□ skill_tree 是否有 4 個核心技能 + 3 個進階技能？
□ skill_tree 的技能名稱是否 15 歲學生能理解？
□ milestone_quests 是否從簡單到困難排列？
□ achievements 是否有 bronze → silver → gold → platinum 的遞進？
□ 是否有台灣產業/企業/現況的連結？
□ 所有中文內容是否使用繁體中文（非簡體）？
□ 是否避免了「了解」「學習」等模糊動詞？（改用「診斷」「設計」「分析」「評估」）
□ YAML 語法是否正確？（用 yaml.safe_load 驗證）
```

### 品質門檻

- 職業 YAML 總分 ≥ 35/50 才能上線
- 起始場景（15 分）必須 ≥ 12 分
- 在地連結（5 分）必須 ≥ 3 分
